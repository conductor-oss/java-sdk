# Scheduled Reports in Java Using Conductor :  Data Query, Report Formatting, and Distribution

A Java Conductor workflow example for scheduled report generation .  querying data sources, formatting the results into a professional report, and distributing to recipients via email or other channels.

## The Problem

You generate regular reports .  daily sales summaries, weekly performance dashboards, monthly compliance reports. Each report requires querying data sources (databases, APIs, data warehouses), formatting the raw data into a readable report (tables, charts, executive summary), and distributing to the right recipients. If the data query fails, you send an empty report. If distribution fails, nobody gets it.

Without orchestration, reports are generated by scripts that query, format, and email in one monolithic block. When the database is slow, the report times out. When the email step fails, nobody knows whether the data was queried successfully. Adding a new recipient or changing the format means modifying the script.

## The Solution

**You just write the data queries and report formatting logic. Conductor handles the query-format-distribute pipeline, retries when data sources are slow, and tracking of data freshness and delivery status for every generated report.**

Each reporting step is an independent worker .  data querying, report formatting, and distribution. Conductor runs them in sequence: query the data, format the report, then distribute it. If the data query fails, Conductor retries. If distribution fails, the report is already generated and can be resent. Every report generation is tracked with data freshness, format, and delivery status. You get all of that, without writing a single line of orchestration code.

### What You Write: Workers

Three workers produce each report: QueryDataWorker fetches data from configured sources, FormatReportWorker structures results into tables and summaries, and DistributeReportWorker delivers the finished report to recipients via email or other channels.

| Worker | Task | What It Does |
|---|---|---|
| **DistributeReportWorker** | `sch_distribute_report` | Distributes a report to recipients. |
| **FormatReportWorker** | `sch_format_report` | Formats a report from queried data. |
| **QueryDataWorker** | `sch_query_data` | Queries data sources for a report. |

Workers simulate scheduled operations with realistic outputs so you can see the scheduling pattern without external systems. Replace with real job logic .  the schedule triggers, retry behavior, and monitoring stay the same.

### The Workflow

```
sch_query_data
    │
    ▼
sch_format_report
    │
    ▼
sch_distribute_report

```

## Running It

### Prerequisites

- **Java 21+**: verify with `java -version`
- **Maven 3.8+**: verify with `mvn -version`
- **Docker**: to run Conductor

### Option 1: Docker Compose (everything included)

```bash
docker compose up --build

```

Starts Conductor on port 8080 and runs the example automatically.

If port 8080 is already taken:

```bash
CONDUCTOR_PORT=9090 docker compose up --build

```

### Option 2: Run locally

```bash
# Start Conductor
docker run -d -p 8080:8080 -p 1234:5000 orkesio/orkes-conductor-standalone:1.2.3

# Wait for Conductor to be ready
until curl -sf http://localhost:8080/health > /dev/null; do sleep 2; done

# Build and run
mvn package -DskipTests
java -jar target/scheduled-reports-1.0.0.jar

```

### Option 3: Use the run script

```bash
./run.sh

# Or on a custom port:
CONDUCTOR_PORT=9090 ./run.sh

# Or pointing at an existing Conductor:
CONDUCTOR_BASE_URL=http://localhost:9090/api ./run.sh

```

## Configuration

| Environment Variable | Default | Description |
|---|---|---|
| `CONDUCTOR_BASE_URL` | `http://localhost:8080/api` | Conductor server URL |
| `CONDUCTOR_PORT` | `8080` | Host port for Conductor (Docker Compose only) |

## Using the Conductor CLI

Start the app in **worker-only mode** so workers keep polling while you use the CLI:

```bash
java -jar target/scheduled-reports-1.0.0.jar --workers

```

Then in a separate terminal:

```bash
conductor workflow start \
  --workflow scheduled_reports_402 \
  --version 1 \
  --input '{"reportType": "standard", "dateRange": "2026-01-01T00:00:00Z", "recipients": "sample-recipients"}'

```

### Check workflow status

```bash
conductor workflow status <workflow_id>
conductor workflow get-execution <workflow_id> -c
conductor workflow search -w scheduled_reports_402 -s COMPLETED -c 5

```

## How to Extend

Each worker handles one reporting step .  connect the data querier to your data warehouse (Snowflake, BigQuery), the distributor to email or Slack, and the query-format-distribute workflow stays the same.

- **DistributeReportWorker** (`sch_distribute_report`): send via SendGrid/SES for email, upload to S3/Google Drive, post to Slack channels, or publish to Confluence
- **FormatReportWorker** (`sch_format_report`): generate PDF reports with Apache PDFBox, Excel spreadsheets with Apache POI, or HTML dashboards with charts
- **QueryDataWorker** (`sch_query_data`): query real data sources. SQL databases (Postgres, MySQL), data warehouses (Snowflake, BigQuery, Redshift), or REST APIs

Swap in your real data warehouse queries and email delivery, and the report generation pipeline runs on schedule without any workflow modifications.
